# AGENTS.md

## 1) 目标与适用范围
- 本文件用于约束本仓库内代理（Agent）执行任务时的默认协作流程。
- 适用范围：仓库根目录及全部子目录。
- 指令优先级：`System/Developer 指令 > 用户当前明确指令 > 本文件`。
- 当高优先级指令与本文件冲突时，按高优先级执行，并在回执中说明偏离原因。

## 2) 默认工作流
- 开始前先执行 `git status --short`，确认当前基线。
- 开始实现前，先在 Notion 查阅 API 清单页，优先复用已有接口，避免重复造轮子：
  - `https://www.notion.so/31cba63208ec810ebdebd78d08036fe3`
- 若 Notion 清单与代码不一致，先更新 Notion 再改代码，并在回执中说明差异。
- 实施完成后做最小可行验证（按变更范围选择）：
  - 后端改动：至少执行相关 `go test` 或 `go test ./...`。
  - Flutter 改动：至少执行相关 `flutter analyze`。
  - 文档改动：做内容与结构自检，确认规则可执行、无歧义。
- 验证通过后，默认执行一次提交（一个需求一个提交）。

## 3) Git 提交规则（核心）
- 默认规则：每完成一个用户需求，提交一次 Git。
- 标准步骤：
  1. `git add -A`
  2. `git status --short` 复核暂存内容
  3. `git commit -m "<message>"`
- 提交信息格式：
  - `feat: ...`
  - `fix: ...`
  - `chore: ...`
- 禁止行为（除非用户明确要求）：
  - `git reset --hard`
  - `git checkout -- <file>`
  - `git commit --amend`
  - 其他会破坏历史或回滚未确认改动的命令

## 4) 产物与噪音文件策略（硬性）
- 默认禁止提交以下内容（除非用户明确要求）：
  - 日志类：`*.log`、`data/logs/`
  - 数据库运行时文件：`*.db-shm`、`*.db-wal`
  - 构建产物：`build/`、`dist/`、平台临时输出目录
- 若变更混入上述文件：
  - 从暂存区移除后再提交（例如 `git restore --staged <path>`）。
  - 如确需长期忽略，先征得用户同意，再补充 `.gitignore`。

## 5) 例外与覆盖
- 以下情况可不自动提交：
  - 用户明确说“不要提交”。
  - 验证未通过（需先反馈失败原因）。
  - 任务仍在中间态，尚未达到可交付标准。
- 若用户明确要求“提交所有更改”，按用户要求覆盖默认粒度执行。

## 6) 回执要求
- 每次提交后必须反馈：
  - `commit hash`
  - `commit message`
  - 当前是否仍有未提交改动（`git status --short` 结果）

## 7) 验收场景（执行对照）
- 普通需求（改多个文件）：预期完成后仅一次提交，提交说明清晰。
- 需求中产生日志/构建目录：预期这些文件不进入最终提交。
- 用户说“先别提交”：预期跳过自动提交并在回执中说明。
- 验证失败：预期不提交，先反馈失败点与建议。
- 用户要求“提交所有更改”：预期允许覆盖默认规则并照做。

## 8) 默认假设与约定
- 文档语言：中文。
- 命名约定：使用本文件 `AGENTS.md` 作为仓库级唯一规范入口。
- 默认提交策略：每个需求一次提交。
- 默认产物策略：运行与构建产物不提交。

## 9) Notion 任务循环流程（执行规范）
- 任务来源：`Self-Study-Tool` 数据库看板。
- 任务优先级（高到低）：`In-Progress > REJECT > TODO > 未开始`。
- 执行顺序（强制）：
  1. 优先处理已有 `In-Progress` 任务直到完成。
  2. 若无 `In-Progress`，处理 `REJECT` 任务并依据驳回原因修复。
  3. 若无 `REJECT`，处理 `TODO` 任务。
  4. 若无 `TODO`，从 `未开始` 领取任务并改到 `TODO`。
  5. 在 Notion 任务页内拆解为可执行小步（不再维护本地 TODO 文件）。
  6. 每次只提取一个小步到 `In-Progress` 后开始编码。
  7. 小步完成并通过最小验证后，更新任务状态。
  8. 全部小步完成后，任务进入 `Review`，等待用户验收。
  9. 验收通过后改为 `Finish`。
- 驳回处理：
  - `Review` 中被驳回的任务，由用户改为 `REJECT` 并写明原因。
  - 代理收到后，优先读取任务页中的驳回原因并逐条修复，再回到 `Review`。
- 一致性要求：
  - 状态流转必须遵循：`未开始 -> TODO -> In-Progress -> Review -> (Finish | REJECT)`。
  - 在 `In-Progress/REJECT/TODO/未开始` 仍有任务时，不中断循环，不切换到无关工作。

## 10) TODO.md 状态
- `TODO.md` 已废弃，不再作为任务来源或状态同步文件。
- 任务分解、状态流转、驳回原因统一记录在 Notion。

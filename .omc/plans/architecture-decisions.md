# Architecture Decision Record - Self-Study-Tool Enhancement

**Date:** 2026-03-11
**Status:** Approved for Implementation

---

## Critical Decisions

### 1. AI Model Selection
**Decision:** Use existing AI service with Claude as primary model
**Rationale:**
- Codebase already has `internal/modules/ai` with Client interface
- Supports multiple providers (Claude, Gemini, OpenAI)
- Use Claude for complex tasks (grading, teaching), Gemini for simple tasks (scheduling)
**Cost Strategy:** User-provided API keys, no built-in rate limiting (user responsibility)

### 2. Network Search Provider
**Decision:** Implement custom HTTP search using existing Go http client
**Rationale:**
- MCP exa NOT configured in codebase (critic verified)
- Custom implementation gives full control
- Use DuckDuckGo HTML API (no API key required)
- Fallback: return empty results if search fails
**Implementation:** New `internal/pkg/websearch` package

### 3. Resource Storage
**Decision:** Local SQLite storage only
**Rationale:**
- Consistent with existing architecture (all data in SQLite)
- Files stored as BLOB or file paths in `materials` table
- No cloud dependency = simpler deployment
**Limits:** 100MB per file, 1GB total per user (enforced in code)

### 4. Onboarding Trigger
**Decision:** First launch detection via `onboarding_completed` flag in profile
**Rationale:**
- Check on app startup: if flag is false, show onboarding
- User can skip and resume later (progress saved in `onboarding_state` table)
**Implementation:** Flutter checks flag on main screen init

### 5. Grading Appeals
**Decision:** No appeal mechanism in v1
**Rationale:**
- User can re-submit answer for new grading
- History preserved in `practice_history` table
- Appeal feature deferred to v2
**Workaround:** User can ask teaching agent to explain grading

### 6. Difficulty Validation
**Decision:** AI-generated rubrics with human review option
**Rationale:**
- Initial rubrics generated by AI using structured prompts
- Stored in `difficulty_rubrics` table
- Admin UI (future) can edit rubrics
**Quality:** AI generates 10-level rubric aligned with Chinese Gaokao standards

### 7. Prompt Engineering Approach
**Decision:** Structured JSON prompts in `prompt_templates` table
**Rationale:**
- Each agent type has dedicated prompt template
- Templates include: system role, task description, output format, examples
- Runtime variables injected: {user_name}, {subject}, {difficulty}, etc.
**Storage:** SQLite table with versioning

### 8. Module Naming
**Decision:** Use `profile` (not `user_profile`)
**Rationale:** Verified in codebase - module is `internal/modules/profile`
**Table Name:** Verify actual table name is `user_profiles` (plural)

### 9. Migration Strategy
**Decision:** Start at migration 024, check for conflicts before each phase
**Rationale:**
- Latest migration is 023
- Before each phase, run `git pull` and verify migration number
- If conflict, increment all subsequent migrations
**Process:** Automated check in implementation script

### 10. Head Teacher Agent Initialization
**Decision:** Singleton service in `internal/modules/agent` package
**Rationale:**
- One head teacher per user (stored in `agents` table with type='head_teacher')
- Created on first onboarding completion
- Persists across sessions
**Lifecycle:** Never deleted, can be reset by user

## Implementation Constraints

1. **Minimal Code:** Each feature implements only core requirements
2. **Git Commits:** One commit per logical unit (35 total planned)
3. **Testing:** Go tests for backend, Flutter widget tests for UI
4. **No External Services:** All features work offline except web search
5. **SQLite Only:** No Redis, no cloud storage, no message queues

---

## Risk Mitigation

1. **AI Cost Overrun:** User provides own API keys, app shows token usage
2. **Search Failures:** Graceful degradation (empty results, user notified)
3. **Large Files:** 100MB limit enforced, user warned before upload
4. **Prompt Quality:** Start with good templates, iterate based on user feedback
5. **Migration Conflicts:** Check before each phase, adjust numbers if needed

## AI Service Integration Strategy

### Implementation:
1. **Use existing AI service:** `internal/modules/ai/service.go`
2. **Client interface:** Already supports Claude, Gemini, OpenAI
3. **Configuration:** User provides API keys in app settings
4. **Prompt execution:**
   ```go
   promptTemplate := promptStore.Get("template_name")
   prompt := promptTemplate.Render(variables)
   response := aiService.Chat(prompt)
   ```
5. **Error handling:** Retry once on timeout, return error to user if fails
6. **Token management:** No built-in limits, user responsibility

### Web Search Alternative:
- **Remove web search requirement** for v1 (optional feature)
- **Fallback:** AI generates questions without external validation
- **Future:** Add legal API (Brave Search) in v2 if needed
